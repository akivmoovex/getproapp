# SQLite → Supabase PostgreSQL (data migration) — **historical**

> **Retirement:** The runnable tooling (`scripts/migrate-sqlite-to-pg/**`, `verify-sqlite-pg-counts.js`, callback SQLite backfill/verify scripts) and the **`better-sqlite3`** dependency were **removed** from this repository. Nothing below is executable on current `main` without recovering old files from **Git history**. The **current** app is PostgreSQL-only (`server.js`).

The rest of this document describes **how that removed tooling behaved**, for archival understanding only.

## Prerequisites (historical operator setup)

1. A **SQLite** file at `SQLITE_PATH` or `data/getpro.sqlite` with columns matching what the old scripts expected.
2. **PostgreSQL** connection string: `DATABASE_URL` or `GETPRO_DATABASE_URL`.
3. **Schema** on Supabase: `db/postgres/000_full_schema.sql` on an empty `public` schema (or drop conflicting objects first).

## 2. Per-step behavior (summary)

- Explicit **IDs** where tables used `GENERATED BY DEFAULT AS IDENTITY`; canonical `tenants.id` values.
- **Timestamps:** SQLite text → `TIMESTAMPTZ`.
- **Booleans:** `0`/`1` → `BOOLEAN`.
- **Idempotency:** `ON CONFLICT … DO NOTHING` where applicable.
- **Verification:** row-count checks between SQLite and PG after writes (non–dry-run).
- **Sequence reset:** `resetIdentitySequence()` after bulk inserts with explicit ids.

## 5. Verification (historical)

The removed **`verify-sqlite-pg-counts.js`** script compared `COUNT(*)` per table between SQLite and PostgreSQL, exiting **1** on mismatch.

## 6. Rollback notes (still valid for PostgreSQL data ops)

PostgreSQL has no automatic undo for a bulk copy. Options:

- **Before migration:** snapshot or empty the `public` schema.
- **After migration:** delete child-first, or `TRUNCATE … CASCADE` on a **dev** database only.

Example `TRUNCATE … CASCADE` block: see earlier revisions of this file in Git history if you need the full table list verbatim.

## 7. Identity / sequence notes (historical)

Steps used `resetIdentitySequence()` after inserts with explicit ids. `tenants` used a plain integer PK without identity.
